## 3. 领域模型

| 文件 | 重点字段 |
|------|----------|
| `Ledger.kt` | id, name, cover_key, deleted_at, created_at, updated_at |
| `Account.kt` | id, name, type, icon_key, card_number, note, balance_minor, include_in_total_assets, deleted_at, updated_at |
| `AccountBalanceRecord.kt` | account_id, date, balance_minor, delta_minor, created_at |
| `Transaction.kt` | id, ledger_id, account_id, category_id, amount_minor, type, date, note, is_excluded, external_source, external_id, deleted_at, updated_at |
| `Category.kt` | id, ledger_id, parent_id, name, icon_key, type, deleted_at, updated_at |
| `Tag.kt` | id, ledger_id, name, deleted_at, updated_at |
| `TransactionTag.kt` | transaction_id, tag_id |
| `Rule.kt` | id, ledger_id, conditions, actions, priority, deleted_at, updated_at |
| `CategoryMemory.kt` | ledger_id, memory_key, category_id, updated_at |
| `RawTransaction.kt` | parser output before rules and dedupe |
| `ImportPreviewItem.kt` | preview row with rule result, dedupe status and user edits |
| `Reminder.kt` | type, amount_minor, schedule fields, paused, deleted_at, updated_at |
| `AppPreference.kt` | key, value, updated_at |
| `SyncState.kt` | idle, syncing, success, error |
| `AppError.kt` | code, message, recoverable |
| `ChartData.kt` | chart type, range, series, labels, empty state, drilldown metadata |

`assets/icons/` 是项目内置 SVG 图标的唯一源目录。图标从 [Lucide](https://lucide.dev/license) 官方静态 SVG 资源中筛选并随项目发布，不在运行时下载。`icon_key` 对应文件名，账户和一级分类均使用该字段。

默认账户初始化为现金类型、项目内置 SVG `icon_key`、`include_in_total_assets = true`。账户余额采用混合模式：收入交易增加余额、支出交易减少余额；不计入收支的交易仍更新余额。编辑或删除交易时按新旧交易差额更新 `Account.balance_minor`，手动校准写入 `AccountBalanceRecord` 且不创建交易。

首版货币固定为人民币。所有金额字段以 `Long` 分为单位存储、计算和传输；仅在平台 UI 按人民币格式化显示。`AccountSummary` 只聚合未删除且 `include_in_total_assets = true` 的账户：正余额为资产，负余额绝对值为负债，净资产为资产减负债。

账户删除使用软删除：删除后不再出现在新增交易的账户选择或资产汇总中，历史交易仍保留 `account_id` 并可查询。

一级分类的 `icon_key` 为非空的项目内置 SVG 图标标识；二级分类的 `icon_key` 可以为空。SVG 图标资源随项目发布，三个客户端均按同一 `icon_key` 渲染，不依赖 SF Symbols 或 Material Icons。

`RawTransaction` / `ImportPreviewItem` 可保留商户名、交易对方、商品说明用于规则和去重；`Transaction` 不保存这些独立字段，确认入账时统一合并进 `note`。

分类删除使用软删除：删除后不再出现在新增交易的分类选择中，历史交易仍可读取原分类并参与搜索和统计。一级分类有未删除的二级分类时拒绝删除。

账本删除使用级联软删除：将账本、其分类、标签、规则和交易标记为删除。默认查询排除这些记录，完整备份仍包含这些记录。

`AppPreference` 存储非敏感、可恢复的应用偏好，包括默认账本、首页账本范围、统计账本范围、交易明细展示模式、界面外观模式、同步目标的非敏感配置和最大备份记录数。WebDAV 凭据、生物识别授权和通知权限不写入该表。删除默认账本时清空默认账本偏好。

## 4. Use Case 与 Facade

目录：

```text
domain/usecase/
├── ledger/
├── account/
├── transaction/
├── import/
├── rule/
├── analytics/
├── search/
├── reminder/
├── sync/
└── interop/
```

关键约定：

- 平台优先调用 Facade / Workflow；底层 Use Case 由 Facade / Workflow 编排。
- Use Case 返回 `Result<T>`、`Flow<T>` 或 `Flow<Result<T>>`，错误通过 `AppError` 传递。
- 需要长期状态的流程用 Flow 暴露进度和结果。

### 4.1 导入 Workflow

```kotlin
interface ImportWorkflow {
    fun preview(request: ImportRequest): Flow<Result<ImportPreviewState>>
    suspend fun editItem(edit: ImportPreviewEdit): Result<ImportPreviewState>
    suspend fun commit(sessionId: ImportSessionId): Result<ImportCommitResult>
}
```

`ImportRequest` 包含目标账本、文件名和文件字节；一个导入会话只处理一个账单文件。解析器先按文件格式和内容识别来源，无法唯一识别时返回待用户选择的候选来源。`ImportPreviewState` 包含解析进度、预览项、重复状态和本次入账笔数、收入合计、支出合计。

内部步骤：

```text
CreateImportPreviewUseCase
  -> ParseBillFileUseCase
  -> ApplyRulesUseCase
  -> ApplyCategoryMemoryUseCase
  -> ApplyDedupeUseCase
  -> SaveImportSessionUseCase

EditPreviewItemUseCase
CommitImportUseCase
```

`ImportSession` 用于预览到入账之间的临时状态。确认入账后清理上传文件、原始解析快照和导入批次记录。

### 4.2 同步 Facade

```kotlin
interface SyncFacade {
    fun observeSyncState(): Flow<SyncState>
    suspend fun configure(config: SyncConfig): Result<Unit>
    suspend fun syncNow(): Result<SyncResult>
}
```

### 4.3 搜索、图表与规则

`Rule` 保存一个结构化匹配条件、一个结构化执行结果和优先级。条件为备注包含关键字、交易类型相等或来源平台相等之一；结果为设置一级分类并可选设置二级分类、设为不计入收支或排除不入账之一。规则按优先级升序执行，命中后短路。

```kotlin
interface SearchTransactionsUseCase {
    suspend operator fun invoke(query: TransactionSearchQuery): Result<SearchResult>
}

`TransactionSearchQuery` 支持账本范围、关键词、交易类型、一级分类、二级分类、标签、账户、金额条件和日期条件的组合。关键词匹配备注、分类名称、账户名称和标签名称。`SearchResult` 包含按交易时间倒序排列的交易列表及总收入、总支出。

interface AnalyticsFacade {
    fun observeDashboard(query: AnalyticsQuery): Flow<Result<AnalyticsDashboardState>>
    suspend fun trend(range: DateRange, granularity: TimeGranularity): Result<ChartData>
    suspend fun categoryShare(query: AnalyticsQuery): Result<ChartData>
    suspend fun accountSummary(): Result<AccountSummary>
    suspend fun periodCompare(query: PeriodCompareQuery): Result<PeriodCompareResult>
    suspend fun tagSummary(query: AnalyticsQuery): Result<TagSummary>
    suspend fun statementTable(query: StatementTableQuery): Result<StatementTable>
}
```

`AnalyticsDashboardState` 包含账本范围、时间范围、收支趋势、收支与结余汇总、上一个等长时间范围的变化、支出与收入排行榜、分类占比和标签分析入口数据。排行榜按金额降序排列。分类占比支持一级分类汇总和一级分类下的二级分类明细两种粒度。`StatementTable` 包含按月收入、支出、结余及全年合计。

统计页平台 ViewModel 维护账本选择列表展开状态、时间范围选择、排行榜收支切换和分类占比粒度。每次启动以当前月初始化时间范围。统计账本范围由 `AppPreference` 独立持久化，首次启动默认所有账本；不与首页账本范围联动。

搜索页平台 ViewModel 维护输入中的关键词、高级筛选展开状态和查询结果。默认账本范围为所有账本；查询状态不持久化，也不与首页或统计页范围联动。手机端以可收起的筛选区呈现，macOS 使用常驻筛选栏。

更多页是二级功能的导航容器。平台 ViewModel 从 KMP 获取全局 `AccountSummary` 和 `SyncState`，用于净资产概览和数据管理状态；其余入口只负责导航到对应模块。应用锁、界面外观模式和数据管理入口之外的设置不在首版实现；界面外观模式由 `AppPreference` 维护。

### 4.4 首页 Facade

```kotlin
interface HomeFacade {
    fun observeHome(query: HomeQuery): Flow<Result<HomeState>>
    fun observeTransactionDetails(query: TransactionDetailQuery): Flow<Result<TransactionDetailState>>
}
```

`HomeState` 包含当前账本范围、当前月份、月度总支出、月度总收入、净收入、日历每日汇总和当前月份全部交易按日期、时间倒序聚合的明细。首页首版不提供预算、趋势或建议入口。月度汇总始终按当前账本范围计算。日历筛选状态仅用于渲染日历每日汇总，不参与月度汇总或明细查询。平台 ViewModel 在每次启动时以当前月初始化月份；账本选择列表的展开状态由平台维护。“所有账本”为固定可选范围。首页账本范围由 `AppPreference` 持久化，首次启动默认所有账本；选择范围后关闭列表并重新订阅对应范围的首页数据。没有可用账本时返回空状态并显示创建账本引导。

`TransactionDetailState` 包含账本范围、日期范围、收入/支出合计及笔数和交易明细。首页平台 ViewModel 维护一个展示模式，供首页内嵌明细和日期明细面板共用。各平台实现一个 `TransactionDetailView`，供两处复用。点击日历日期时，平台 ViewModel 以该日期和当前首页账本范围创建独立状态；面板内更改范围只更新该状态，不改写首页明细状态。手机端以底部弹出面板显示，macOS 以窗口内详情面板显示。日期面板新增交易时预填选中日期；单账本范围预填该账本，所有账本范围不预填账本。

### 4.5 记账页状态

记账页平台 ViewModel 负责维护 UI-only 状态：当前交易类型切换、选中一级分类、选中二级分类、选中账户、选中标签、备注、日期、数字键盘输入值、提交按钮状态。统一的 `AddTransactionContext` 可传入预填日期和账本；无预填时使用当前时间、默认账本（如存在）和现金账户，初始类型为支出。默认账本是可为空的应用偏好；不存在时账本保持未选择，保存前必须选择。无可用账本或账户时显示对应的创建引导。一级分类图标网格和二级分类快捷选项来自 KMP 分类数据。保存再记保留类型、账本、账户和日期，清空金额、分类、标签和备注。保存动作调用 KMP 的交易 Use Case。

## 5. 解析器设计

### 5.1 commonMain Interface

```kotlin
expect class CsvDecoder {
    fun decode(bytes: ByteArray, charset: CsvCharset): String
}

interface BillParser {
    fun parse(bytes: ByteArray): Result<List<RawTransaction>>
}

expect class SpreadsheetParser() {
    fun parseXlsx(bytes: ByteArray): Result<List<RawTransaction>>
    fun parseXls(bytes: ByteArray): Result<List<RawTransaction>>
}
```

CSV 行列解析逻辑放在 commonMain；字符集转换和 spreadsheet 读取使用 expect/actual。

### 5.2 平台实现

| 来源 | commonMain | Android | Apple |
|------|------------|---------|-------|
| 支付宝 CSV/GBK | 行列解析共享 | JVM `Charset.forName("GBK")` | Foundation GB18030 |
| 京东 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 美团 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 抖音 CSV/UTF-8 BOM | 完全共享 | 同 commonMain | 同 commonMain |
| 青子记账 JSON | 完全共享 | 同 commonMain | 同 commonMain |
| 微信 XLSX | Parser Interface | Apache POI | CoreXLSX |
| 建设银行 XLS/BIFF8 | Parser Interface | Apache POI | 极简 BIFF8 parser |

Apple GBK 解码使用 Foundation 的 GB18030 编码能力，GB18030 是 GBK 超集。

### 5.3 建设银行 XLS 重点实现

基于真实 CCB.xls 样例：

- 全部字段均为纯文本字符串，无数字类型、日期序列号、公式、合并单元格。
- 金额以带千分位字符串存储，去逗号后转金额。
- 日期为 `yyyyMMdd` 文本。
- 结构固定：前 3 行元数据，第 4 行表头，第 5 行起数据。

Apple 极简 BIFF8 parser 只解析：

- SST record `0x00FC`
- LabelSST record `0x00FD`

Apple 首版 BIFF8 parser 的实现范围限定为当前建设银行 fixture 所需记录类型。升级条件：真实样例出现数字、公式、合并单元格或其他未覆盖记录类型。

## 6. 规则、记忆与去重

### 6.1 规则

- Rule conditions 存储为结构化条件列表，规则系统不执行自由文本脚本。
- 首版一条规则只包含一个条件和一个结果。
- 条件支持：备注包含关键字、交易类型相等、来源平台相等。
- 结果支持：设置一级分类并可选设置二级分类、设为不计入收支、排除不入账。
- 同账本内按 priority 升序执行，命中后短路。

### 6.2 分类记忆

- 分类记忆以 `ledger_id + memory_key` 为主键。
- `memory_key` 的生成规则首版保持简单：来源平台 + 规范化备注。
- 用户在导入预览页手动选择分类后写入分类记忆。
- 分类记忆只在规则未命中时使用。

### 6.3 去重

- 绝对去重查询 `external_source + external_id`；青子记账导入不作为 `external_source`。
- 模糊去重条件：时间窗口 + 金额相等 + 备注文本相似度超过阈值。
- 普通来源时间窗口为前后 2 小时。
- 建设银行来源窗口为同日。
- 疑似重复默认排除入账。

### 7.2 Adapter Interface

```kotlin
interface SyncAdapter {
    suspend fun listBackups(): Result<List<RemoteBackupMeta>>
    suspend fun uploadBackup(backup: BackupRecord): Result<Unit>
    suspend fun downloadBackup(meta: RemoteBackupMeta): Result<BackupRecord>
    suspend fun deleteBackup(meta: RemoteBackupMeta): Result<Unit>
}
```

### 7.3 备份记录

```kotlin
data class BackupRecord(
    val deviceId: String,
    val backupId: String,
    val createdAt: Instant,
    val payload: String
)
```

- `payload` 为序列化后的完整业务数据和应用偏好快照。WebDAV 凭据、生物识别授权和通知权限不写入 `payload`。
- 首版备份记录上传前不加密。
- 远端文件名：`backups/{created_at}-{device_id}.backup`。
- `SyncMeta.sq` 记录最近备份时间和本机同步执行状态；同步目标的非敏感配置和最大备份记录数存储在 `AppPreference`。
- 恢复流程先由用户确认；确认后以备份 `payload` 覆盖当前可恢复状态，不合并本地数据。确认界面可先调用 `syncNow()` 创建当前备份。
- 已配置同步目标且网络可用时，可恢复状态写入后启动防抖自动备份；连续写入合并为一次备份。手动 `syncNow()` 立即执行并合并待执行的自动备份。

- 调用 KMP Facade / Use Case。
- 将 KMP Flow / async 结果映射成 SwiftUI 状态。
- 处理平台导航、窗口命令、文件选择、本地通知和 Keychain。

iOS 使用 `Home / Analytics / AddTransaction / Search / More` 五个 Tab。macOS 使用同一信息架构的 Sidebar。`More/` 下承载 Settings、Import、Export、Rules、Reminders、Ledgers、Accounts、Assets、Category Management、Tag Management。

## 10. 命名与依赖规则

| 类型 | 规范 | 示例 |
|------|------|------|
| Kotlin Entity | PascalCase + 无后缀 | `Transaction.kt` |
| Kotlin Use Case | PascalCase + `UseCase` | `AddTransactionUseCase.kt` |
| Kotlin Facade / Workflow | PascalCase + `Facade` / `Workflow` | `ImportWorkflow.kt` |
| Repository Interface | PascalCase + `Repository` | `TransactionRepository.kt` |
| Repository Implementation | PascalCase + `RepositoryImpl` | `TransactionRepositoryImpl.kt` |
| SQLDelight schema | PascalCase + `.sq` | `Transaction.sq` |
| Swift View | PascalCase + `View` | `TransactionListView.swift` |
| Swift ViewModel | PascalCase + `ViewModel` | `TransactionViewModel.swift` |
| Android Screen | PascalCase + `Screen` | `TransactionListScreen.kt` |
| Android ViewModel | PascalCase + `ViewModel` | `TransactionViewModel.kt` |

依赖规则：

- `domain/` 的依赖范围限定为领域模型和 Repository Interface。
- `data/` 的依赖范围限定为领域模型、Repository Interface、存储和同步实现。
- Platform UI 通过 KMP Facade / Use Case 访问业务能力。
- SQLDelight Query 的调用位置限定在 KMP 数据层。
- 解析器、同步和导入操作必须在后台线程执行。
