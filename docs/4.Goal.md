# OmniFlow Goal

本文档定义阶段性验收目标。阶段顺序即推荐实现顺序；每一条目标都必须可验证。

---

## 阶段 0：文档与工程骨架

### 文档验收

- [ ] `docs/1.Requirements.md` 只描述产品需求和用户可见行为。
- [ ] `docs/2.Architecture.md` 只描述技术架构、模块职责、分层规则和关键技术路线。
- [ ] `docs/3.TechDesign.md` 只描述代码目录结构、关键 Interface 和重点实现细节。
- [ ] `docs/4.Goal.md` 按阶段定义验收目标。
- [ ] 文档引用统一指向 `docs/1.Requirements.md`、`docs/2.Architecture.md`、`docs/3.TechDesign.md`、`docs/4.Goal.md`。

### 工程结构验收

- [ ] 顶层目录包含 `docs/`、`examples/`、`shared/`、`androidApp/`、`appleApp/`、`build.gradle.kts`、`settings.gradle.kts`、`gradle.properties`。
- [ ] `docs/` 至少包含 `1.Requirements.md`、`2.Architecture.md`、`3.TechDesign.md`、`4.Goal.md`。
- [ ] `examples/` 包含账单解析样例及首页、统计、记账、搜索、更多、设置的 UI 参考图片。
- [ ] `assets/icons/` 包含随项目发布的统一 SVG 图标资源。
- [ ] `shared/` 是 KMP 共享模块，包名根路径为 `com.omniflow.shared`。
- [ ] `androidApp/` 是 Android 原生工程，包名根路径为 `com.omniflow.android`。
- [ ] `appleApp/` 是一个 Apple 工程，包含 iOS 和 macOS 两个 Target。

### 架构边界验收

- [ ] 项目主线采用 KMP 共享领域层 + 平台原生 UI。
- [ ] 正式用户端技术栈限定为 SwiftUI、Jetpack Compose 和 KMP。
- [ ] Android UI 使用 Kotlin + Jetpack Compose + Material 3。
- [ ] Apple UI 使用 SwiftUI。
- [ ] Apple 端通过 Kotlin/Native Framework 接入 KMP，并使用 SKIE 或等价方案桥接 `suspend fun` / `Flow`。
- [ ] ViewModel 全部位于平台侧，KMP 暴露业务 Facade / Use Case / Flow。
- [ ] 平台 UI 只通过 KMP 暴露的 Use Case、Facade 和 Flow 访问数据和业务能力。
- [ ] 解析、规则、去重、统计和同步逻辑集中在 KMP 层。

## 阶段 1：共享领域层与本地数据基础

### KMP 分层验收

- [ ] `domain/` 的依赖范围限定为领域模型和 Repository Interface。
- [ ] `data/` 的依赖范围限定为领域模型、Repository Interface、存储和同步实现。
- [ ] Repository Interface 放在 domain 层，Repository Implementation 放在 data 层。
- [ ] KMP 面向平台暴露 Use Case / Facade / Flow。
- [ ] Use Case / Facade 返回 `Result<T>`、`Flow<T>` 或 `Flow<Result<T>>`。
- [ ] 错误通过 `AppError` 或等价错误模型传递，包含稳定错误码、用户可读消息和可恢复性标记。

### 数据存储验收

- [ ] 本地 SQLite 是每台设备的主数据存储。
- [ ] SQLDelight `.sq` 文件定义 schema 和 migration。
- [ ] SQLDelight 查询只在 KMP 数据层内使用。
- [ ] 所有实体主键使用 UUID 字符串。
- [ ] Ledger / Account / Transaction / Category / Tag / Rule / Reminder 等实体支持软删除。
- [ ] 交易表包含 `ledger_id`。
- [ ] 账户表不包含 `ledger_id`。
- [ ] 分类、标签、规则、分类记忆均包含 `ledger_id`。
- [ ] 标签使用 `TransactionTag` 关联表。
- [ ] `AppPreference` 保存非敏感、可恢复的应用偏好并参与备份恢复。
- [ ] 所有金额和余额均以分为单位存储和计算，不使用浮点数。

### 基础查询验收

- [ ] 所有交易列表查询默认按账本过滤。
- [ ] 所有统计查询默认按账本过滤。
- [ ] 所有搜索查询默认按账本过滤。

## 阶段 2：账本、账户、分类与标签

### 多账本验收

- [ ] 首版账户模型为单用户多账本。
- [ ] 支持创建、编辑、删除多个账本。
- [ ] 删除账本后，账本及其分类、标签、规则和交易不再出现在默认查询中，但保留在备份数据中。
- [ ] 每条交易必须归属于一个账本。
- [ ] 首次启动时自动创建一个初始账本，但不自动设为默认账本。
- [ ] 默认账本是可选项；存在默认账本时，未预填账本的记账页自动选择它。
- [ ] 不存在默认账本时，未预填账本的记账页不选择账本，用户必须在保存前选择账本。
- [ ] 删除默认账本后，默认账本自动取消。
- [ ] 没有可用账本时，首页显示创建账本引导；记账和导入必须先创建或选择账本。
- [ ] 账本支持设置名称和封面。
- [ ] 新建账本时仅复制系统预置分类模板。
- [ ] 默认规则由用户按需创建。

### 账户验收

- [ ] 账户为全局共享资源，不从属于任何账本。
- [ ] 支持创建、编辑、删除账户。
- [ ] 账户类型支持：现金、储蓄卡、信用卡、电子钱包、投资账户。
- [ ] 首次启动时自动创建现金、支付宝、微信、京东、美团、抖音、建设银行账户；默认账户类型均为现金。
- [ ] 默认账户使用项目内置 SVG 图标，默认计入总资产。
- [ ] 账户支持设置项目内置 SVG 图标、名称、卡号、备注、余额、是否计入总资产。
- [ ] 删除账户后，不再用于新增交易且不计入资产汇总；历史交易保留原账户并继续参与查询。
- [ ] 账户余额采用混合模式：交易自动影响余额，同时支持用户手动校准当前余额。
- [ ] 手动校准余额保留按天更新记录和增量记录。
- [ ] 手动校准只更新账户余额并保留校准记录，不新增交易。
- [ ] 资产页展示计入总资产账户的余额、资产合计、负债合计和净资产，并可进入账户详情、创建账户和编辑账户。
- [ ] 未删除且计入总资产的账户，正余额计入资产，负余额绝对值计入负债，净资产等于资产减负债；账户类型不改变该规则。
- [ ] 没有可用账户时，记账页显示创建账户引导，用户必须创建或选择账户后才能保存交易。
- [ ] 还款提醒归属于提醒模块，账户模型字段只覆盖账户信息。

### 货币验收

- [ ] 首版仅支持人民币，所有金额、余额、资产和统计结果均以人民币展示。
- [ ] 首版不支持多币种、汇率设置或金额换算。

### 分类与标签验收

- [ ] 分类按账本独立。
- [ ] 分类为两级树结构：一级分类和二级分类。
- [ ] 每个一级分类均有一个项目内置 SVG 图标。
- [ ] 一级分类图标仅可从项目内置的统一 SVG 图标库中选择；二级分类可以没有图标。
- [ ] 系统内置支出一级分类：餐饮、交通、日用、服务、娱乐、医疗、旅游、汽车、住房、数码、服饰。
- [ ] 系统内置收入一级分类：工资、理财、副业、奖金、红包。
- [ ] 每个分类有固定类型属性：收入或支出。
- [ ] 每条已入账交易必须关联一个当前账本内的分类。
- [ ] 删除分类后不再用于新增交易；历史交易保留原分类并继续参与搜索和统计。
- [ ] 一级分类存在未删除的二级分类时，必须先处理其二级分类后才能删除。
- [ ] 标签按账本独立。
- [ ] 标签为扁平多标签系统。
- [ ] 一条交易可以打多个标签，也可以不打标签。
- [ ] 支持按标签筛选和聚合。

## 阶段 3：手动记账闭环

### 交易录入验收

- [ ] 记账页第一版视觉与交互参考 `examples/记账.PNG`。
- [ ] 支持手动新增一条收入或支出记录，包含：金额、分类、账户、备注、标签、日期。
- [ ] 记账页支持支出、收入切换。
- [ ] 记账页以图标网格展示用户自定义的一级分类。
- [ ] 选中一级分类后，记账页展示该一级分类下的二级分类快捷选项。
- [ ] 记账页支持选择账户、标签、日期。
- [ ] 记账页支持备注输入。
- [ ] 记账页使用数字键盘录入金额，支持加、减、小数点、退格。
- [ ] 记账页支持「保存再记」和「完成」两个提交动作。
- [ ] 主导航、首页和日期明细面板的新增入口均打开同一记账页，并传递调用入口的预填日期和账本。
- [ ] 新建交易默认选择支出、当前时间和现金账户；仅在存在默认账本时自动选择默认账本。
- [ ] 一级分类必须选择，二级分类可不选择；选择二级分类时交易关联二级分类，否则关联一级分类。
- [ ] 保存再记保留交易类型、账本、账户和日期，清空金额、分类、标签和备注；完成后关闭记账页。
- [ ] 手动新增交易必须处于一个目标账本上下文内。
- [ ] 手动新增交易保存前必须选择当前账本内的分类。
- [ ] 手动新增交易允许来源平台和外部订单号为空。
- [ ] 交易类型枚举为收入、支出。

### 交易维护验收

- [ ] 支持编辑已有交易的任意字段。
- [ ] 支持删除交易，删除后默认不出现在列表、搜索和统计中。
- [ ] 支持将一笔交易标记为不计入收支，标记后该笔交易在统计图表中不参与计算。
- [ ] 不计入收支的交易仍按交易类型更新账户余额。
- [ ] 新增、编辑或删除交易后，关联账户余额按交易变更同步更新。
- [ ] 预算功能属于首版范围外。

## 阶段 4：账单解析与导入预览

### 导入 Workflow 验收

- [ ] KMP 暴露 `ImportWorkflow` 或等价 Facade。
- [ ] 导入流程由 KMP 编排解析、规则、去重、预览和入账步骤。
- [ ] `ImportWorkflow.preview(...)` 返回可观察的导入预览状态。
- [ ] `ImportWorkflow.editItem(...)` 支持预览项编辑。
- [ ] `ImportWorkflow.commit(...)` 负责确认入账。
- [ ] `ImportSession` 只保存预览到入账之间的临时状态。
- [ ] 确认入账后清理 `ImportSession`、上传文件、原始解析快照和导入批次记录。
- [ ] 解析器输出统一 `List<RawTransaction>`，下游规则引擎和去重引擎无需感知来源差异。
- [ ] 解析器和导入流程在后台线程执行，主线程只接收进度和结果。

### 解析器验收

- [ ] 支付宝解析器能解析 GBK CSV。
- [ ] 微信解析器能解析 XLSX。
- [ ] 京东解析器能解析 UTF-8 BOM CSV。
- [ ] 美团解析器能解析 UTF-8 BOM CSV。
- [ ] 建设银行解析器能解析 Excel 97-2003 XLS。
- [ ] CSV 行列解析逻辑在 commonMain 共享。
- [ ] Android GBK 解码使用 JVM Charset 或等价能力。
- [ ] Apple GBK 解码使用 Foundation GB18030 或等价能力。
- [ ] Android XLSX/XLS 解析使用 Apache POI 或等价稳定方案。
- [ ] Apple XLSX 解析使用 CoreXLSX 或等价纯 Swift 方案。
- [ ] Apple 建设银行 XLS parser 覆盖当前 fixture 所需的 SST 与 LabelSST。
- [ ] 文件格式不匹配时返回明确错误。

### 账单来源验收

- [ ] 支持支付宝 CSV/GBK 导入。
- [ ] 支持微信 XLSX 导入。
- [ ] 支持京东 CSV/UTF-8 BOM 导入。
- [ ] 支持美团 CSV/UTF-8 BOM 导入。
- [ ] 支持建设银行 XLS 导入。
- [ ] 支付宝 fixture `examples/支付宝.csv` 全部 107 条记录解析成功且字段映射正确。
- [ ] 微信 fixture `examples/微信.xlsx` 全部 86 条记录解析成功。
- [ ] 京东 fixture `examples/京东.csv` 全部 16 条记录解析成功。
- [ ] 美团 fixture `examples/美团.csv` 全部 18 条记录解析成功。
- [ ] 建设银行 fixture `examples/CCB.xls` 全部 152 条记录解析成功。

### 通用导入验收

- [ ] 用户从更多页进入导入，先选择目标账本，再通过系统文件选择器选择一个账单文件。
- [ ] 系统根据文件格式和内容识别来源；无法唯一识别时允许用户选择来源。
- [ ] 导入前必须选择目标账本；该导入会话内所有明细固定归属于目标账本。
- [ ] 账户根据导入的账单类型自动解析并填写。
- [ ] 青子记账导入明细默认归属现金账户。
- [ ] 支付宝和京东自带交易分类字段作为来源字段保留；App 分类由规则、历史分类记忆或用户手动选择决定。
- [ ] 微信、美团、建设银行无自带分类字段时，分类由规则或历史分类记忆填充；分类为空的未排除项阻止确认入账。
- [ ] 「不计收支」或「中性交易」类交易自动标记为不计入收支。
- [ ] 来源未提供收支方向的中性交易必须在预览页由用户选择收入或支出后才能确认入账。
- [ ] 交易订单号正确提取为外部订单号，用于后续去重。
- [ ] 建设银行金额正数为收入，金额负数为支出。
- [ ] 建设银行交易时间仅精确到天，解析为当天开始时间。
- [ ] 确认入账后的最终交易必须保留来源平台和外部订单号；青子记账导入不作为来源平台。
- [ ] 撤销上一次导入属于首版范围外。

### 导入预览验收

- [ ] 用户上传文件后，系统自动完成解析、规则匹配、分类记忆和去重检测。
- [ ] 解析过程中展示进度和可读错误；解析失败不写入交易。
- [ ] 预览页展示完整待入账明细列表。
- [ ] 每条预览明细显示：日期、金额、备注、分类、标签、是否排除、来源平台。
- [ ] 重复项和疑似重复项在预览页有醒目标识。
- [ ] 用户可逐条编辑分类、账户、标签、备注、是否排除。
- [ ] 用户可批量全选、反选、改分类、排除。
- [ ] 确认前展示本次入账笔数、收入合计和支出合计。
- [ ] 用户确认后一键入账，所有通过确认的明细写入数据库。

## 阶段 5：规则、分类记忆与去重

### 规则引擎验收

- [ ] 规则按账本独立。
- [ ] 规则页支持切换账本、创建、编辑、删除和调整规则优先级。
- [ ] 一条规则包含名称、一个匹配条件、一个执行结果和优先级。
- [ ] 规则执行时机限定为导入预览阶段。
- [ ] 支持条件：备注包含关键字、交易类型等于收入或支出、来源平台等于指定来源。
- [ ] 支持结果：设置一级分类并可选设置二级分类、设为不计入收支、排除不入账。
- [ ] 规则执行动作限定在当前导入目标账本内。
- [ ] 规则按用户定义的优先级顺序执行，命中后短路。
- [ ] 支持规则创建、读取、更新、删除。
- [ ] 用户可在导入预览页覆盖规则结果，最终以用户确认的预览结果入账。

### 自动分类记忆验收

- [ ] 规则未命中时，系统使用同账本历史分类记忆自动填分类。
- [ ] 历史分类记忆仍未命中时，预览项分类为空。
- [ ] 分类为空且未排除的预览项阻止确认入账。
- [ ] 用户在预览页手动选择分类后，该选择进入同账本历史分类记忆。
- [ ] 历史分类记忆优先级低于用户自定义规则。

### 去重验收

- [ ] 绝对去重：相同来源平台 + 外部订单号的交易直接标记为重复；青子记账导入不作为来源平台。
- [ ] 被标记为绝对重复的明细入账状态锁定为排除。
- [ ] 模糊去重：时间窗口 + 金额相等 + 备注文本相似度超过阈值时标记为疑似重复。
- [ ] 普通来源模糊去重时间窗口为前后 2 小时。
- [ ] 建设银行来源因时间仅精确到天，模糊去重窗口放宽为同日。
- [ ] 疑似重复项默认排除入账；用户手动确认保留后进入可入账状态。

## 阶段 6：搜索、聚合与图表

### 高级搜索验收

- [ ] 搜索页第一版视觉与交互参考 `examples/搜索.PNG`。
- [ ] 搜索默认范围为所有账本，支持筛选单个账本；搜索账本范围不与首页或统计页联动，也不在重启 App 后保持。
- [ ] 支持按关键词、交易类型、一级分类、二级分类、标签、账户、金额和日期任意组合筛选。
- [ ] 关键词匹配备注、一级分类名称、二级分类名称、账户名称和标签名称。
- [ ] 金额支持不限、精确金额、最小金额和最大金额；日期支持不限、某一天和时间范围。
- [ ] 手机端高级筛选默认展开，支持收起、展开和一键清除；macOS 使用常驻筛选栏。
- [ ] 输入关键词或调整筛选条件后更新搜索结果；全部筛选恢复默认时不展示结果。
- [ ] 搜索结果按交易时间倒序展示，点击交易可编辑。
- [ ] 搜索结果页自动计算并展示筛选范围内的总收入和总支出。

### 图表验收

## 阶段 7：提醒与隐私保护

### 提醒验收

- [ ] 提醒与周期事项是独立功能，独立于账本、账户和交易。
- [ ] 提醒模块只产生提醒事件；交易、统计和账户余额由记账模块维护。
- [ ] 提醒与周期事项支持创建、编辑、暂停、删除。
- [ ] 提醒与周期事项参与多端同步。
- [ ] 提醒使用本地系统通知。
- [ ] 提醒分为还款提醒和订阅提醒两个子类型。
- [ ] 还款提醒金额可选，只用于提醒展示。
- [ ] 还款提醒支持固定还款日期。
- [ ] 还款提醒支持根据账单日后 N 天计算还款日。
- [ ] 订阅提醒金额可选，用于展示和未来固定支出概览。
- [ ] 订阅提醒支持每天、每周、每月、每年重复。
- [ ] 复杂 RRULE、法定节假日顺延和工作日规则属于首版范围外。
- [ ] 首版提醒不生成待确认记账卡片，不与导入交易自动合并。

### 隐私保护验收

- [ ] 支持应用锁。
- [ ] 用户启用应用锁后，启动 App 或从后台回到前台时要求解锁。
- [ ] 解锁方式使用平台支持的面容、指纹或系统密码能力。
- [ ] 应用锁保护本机访问。

## 阶段 8：同步与备份恢复

### 同步实现验收

- [ ] KMP 层负责 iCloud / WebDAV 同步、同步 Adapter、备份生成和备份恢复。
- [ ] 同步通过 `SyncAdapter` 或等价 Interface 接入 iCloud / WebDAV，业务层只依赖同步 Adapter Interface。
- [ ] 远端只保存全量备份记录。
- [ ] 首版上传到远端的备份记录不加密。
- [ ] 每个备份记录包含恢复 App 所需的全部业务数据和应用偏好。
- [ ] WebDAV 凭据、生物识别授权和通知权限不写入备份记录；恢复后由当前设备重新配置或授权。
- [ ] 用户可配置最大备份记录数。
- [ ] 最大备份记录数默认值为 10。
- [ ] 超过最大备份记录数时删除最旧备份记录。
- [ ] 平台 UI 可展示备份记录列表、最近备份时间、进行中进度、失败原因和手动重试入口。
- [ ] 恢复确认后，选定备份完整替换当前设备的全部可恢复状态，不与本地数据合并。
- [ ] 恢复确认界面提供创建当前备份的入口。

### 同步功能验收

- [ ] 本地优先：无网络时基础记账、导入预览、搜索和统计分析均可使用。
- [ ] 有网络时支持自动或手动同步。
- [ ] 已配置同步目标且网络可用时，可恢复状态变更后延迟自动创建备份；连续变更合并为一次备份。
- [ ] 用户可随时手动创建备份，手动备份立即执行。
- [ ] 远端职责限定为备份记录文件存储。
- [ ] 用户删除 App 后重新安装，连接同一同步目标并恢复备份记录后，数据与上传前一致。
- [ ] 平台 UI 可展示同步状态、最近同步时间、进行中进度、失败原因和手动重试入口。
