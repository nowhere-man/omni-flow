# iOS UI 审计问题修复设计

日期：2026-07-16  
基线：`master` / `1808b1ab3e974c29e0c4a9e74ae13469ca5ea986`  
实施分支：`codex/fix-ios-ui-audit`

## 1. 背景与目标

当前 iOS 界面已具备首页、统计、搜索、记账和管理等主要流程，但源码审计确认了金额精度、异步保存、导入编辑、日期明细、统计信息结构、无障碍、触控尺寸和小屏布局等问题。本次工作修复所有已有源码或产品要求支持的确定问题，并对已识别的性能热点做低风险收窄。

完成后的可观察结果：

- 金额、导入、提醒、管理编辑器和交易编辑器不会保存错误数据或丢失失败时的输入。
- 日期明细具备独立账本范围、新增入口、笔数、正确状态和与首页一致的展示方式。
- 统计默认按月，具备汇总、对比、趋势、单笔排行、标签分析和可读的年度账单表。
- iOS 导航、Sheet、触控、颜色、Dynamic Type 和 VoiceOver 行为符合 Apple HIG；iOS 26 使用原生 Liquid Glass，并为 iOS 16–25 提供可交互 fallback。
- 平台无关的统计语义和数据模型在 `shared` 中实现，并同步更新 Android 消费端；macOS 保留桌面容器和信息密度。
- 不新增第三方依赖，不进行缺乏 Instruments 证据的 `AppStore` 大规模拆分。

## 2. 范围边界

### 2.1 本次范围

1. iOS 首页、日期明细、主导航和交易新增/编辑。
2. iOS 统计页、排行榜、趋势、标签分析和账单表。
3. iOS 搜索、更多页、设置和管理类编辑器。
4. 导入预览、提醒、规则等涉及共享业务语义的 Apple 状态适配。
5. 共享统计模型、查询和测试，以及受其影响的 Android 统计 UI。
6. Apple 通用控件、语义色、Liquid Glass fallback、SVG 图标缓存和局部状态依赖优化。

### 2.2 明确不做

- 不添加预算、债务、OCR、云服务或其他未在当前产品需求中的新模块。
- 不把 iOS 移动布局照搬到 macOS；macOS 只共享真正相同的数据和控件行为。
- 不引入新的架构框架或第三方 SVG/图表库。
- 没有 Instruments `.trace` 时，不宣称已经消除所有卡顿，也不拆分整个 `AppStore`。
- 不创建 tag、Release 或直接推送 `master`。

## 3. 总体方案

采用“按用户风险分批、共享规则先于平台渲染”的方案：

1. 先修数据正确性和异步编辑生命周期，避免继续产生错误或让用户丢失输入。
2. 再补齐日期明细和交易编辑器的完整闭环。
3. 扩充共享统计模型，同时更新 iOS 与 Android 的统计呈现。
4. 统一无障碍、触控、布局、语义色和导航语义。
5. 最后做不改变业务结果的缓存和失效范围收窄。

该方案比一次性重写页面更容易验证，也能让每个提交对应一组明确验收标准。

## 4. 详细设计

### 4.1 金额、导入和异步保存正确性

#### 金额格式

- 移除会做整数除法的 `wholeRmb` 展示路径，统一使用保留角分的货币格式化器。
- 首页日期明细和日分组汇总都使用同一格式化实现；`1`、`105`、`100` 分分别显示为 `¥0.01`、`¥1.05`、`¥1.00`。
- 金额的正负语义由视图明确添加，格式化器只负责绝对金额和本地化货币文本。

#### 导入预览

- 未识别交易类型使用显式“请选择类型”状态，不再把 `nil` 渲染成“支出”。确认导入前，所有未跳过项目都必须具有合法类型。
- 所有 Toggle 的 setter 使用 SwiftUI 传入的新值，不再无条件反转旧值。
- 导入 session 创建后锁定目标账本。用户选择“更换账本”时先清空当前 session 和预览，再回到选择步骤，防止分类资源与目标账本不一致。
- 行内异步编辑以“项目 ID + 字段 + revision”标识请求，只接受最新 revision 的回调。旧回调可以完成，但不得覆盖更新后的本地值或错误状态。
- 导入错误显示在对应行或当前导入步骤中，保留用户已经完成的其他编辑。

#### 管理编辑器与提醒

- `saveLedger`、`saveAccount`、`saveCategory`、`saveTag`、`saveRule`、`saveReminder` 等保存入口返回明确的成功/失败结果。
- 编辑器在保存期间禁用重复提交，成功后才关闭；失败时继续停留并在编辑器内显示错误，保留全部输入。
- 提醒金额为空表示“不限制金额”；非空但无法解析、为零或为负时阻止保存并给出字段错误。
- 规则来源条件、规则动作、提醒周期和导入重复状态全部映射为面向用户的中文标签，不直接展示内部枚举名或原始错误描述。
- 规则列表根据真实动作显示“分类”“账户”“标签”或其他目标，不再统一写成“→ 分类”。

### 4.2 日期明细与交易编辑闭环

#### 日期明细

- 日期明细使用 `AppStore` 中与首页共享的 `transactionDisplayMode`，任一位置切换后两处保持一致；不再维护独立的默认卡片状态。
- 标题栏提供账本菜单和新增按钮。新增交易默认使用当前明细账本；若范围是单日，则默认使用该日，范围明细则使用范围起始日。
- 明细头部同时显示收入/支出金额和对应笔数。笔数按当前类型过滤和当前账本范围计算。
- 明确区分四种状态：加载中、加载失败、空数据、有数据。失败状态提供重试，不再被渲染为“暂无明细”。
- 切换局部账本只更新 `dateDetailLedgerID` 和日期明细订阅，不改变首页全局账本选择。
- 从明细打开交易详情、编辑后返回以及关闭 Sheet 时，维持现有的延迟呈现机制，避免两个 Sheet 同时竞争。

#### 交易编辑器

- 增加原生 `Toggle("不计入收支")`，编辑时回显、保存时提交 `excluded`。
- 账本变化后清空旧账本的分类和标签选择，再加载新账本资源；初始化已有交易时不触发误清空。
- 账本和账户菜单同时显示图标与当前名称，并为长名称设置单行截断和完整 VoiceOver 标签。
- 编辑器中的分类区域只负责选择，不承担排序。移除长按拖拽；排序继续放在分类管理页面，避免与外层滚动手势冲突。
- 分类选择使用横向滚动的两行 `LazyHGrid`，外层只保留页面级纵向滚动，不再嵌套固定高度的纵向 `ScrollView`。
- 二级分类、标签、图标按钮及清除操作的实际命中区域不小于 44×44pt。

### 4.3 统计与共享数据语义

统计是全产品通用行为，因此模型与查询放在 `shared`，iOS 和 Android 使用各自原生控件呈现。

#### 共享模型与查询

- 默认范围改为“月”。当前范围变化时，同时计算前一个等长范围的汇总，用于支出、收入、结余的变化值；无前期数据时显示“暂无对比”，不显示虚假的 0%。
- 复用现有 `trend` 能力并补齐连续时间点。没有交易的日期/周/月返回零值点，避免折线或柱状图断裂。
- 排行榜模型改为单笔交易排行，包含稳定交易 ID、发生时间、分类显示名、备注、图标、类型和金额。分类聚合仍由分类占比模块负责。
- 增加标签分析模型：按标签汇总金额和笔数，并保留无标签分组。标签分析遵循当前账本、时间范围和收入/支出类型。
- 年度账单表继续由共享层提供月份收入、支出和结余，并支持选定年份；iOS/Android 的筛选仅影响呈现，不重复实现计算规则。
- 为范围边界、前期对比、零值补点、单笔排行、标签汇总、排除交易和多账本过滤增加 JVM 测试。

#### iOS 统计页

- 默认选中“月”，顶部范围选择和日期切换保持原生 `Picker`/`Menu` 语义。
- 首屏汇总卡显示总支出、总收入、总结余和相对前期变化；颜色只是辅助，文字和符号同时表达涨跌。
- 增加通用趋势卡，根据范围选择日/周/月粒度；使用 Swift Charts 并保留系统图表探索能力。
- “收支排行榜”展示单笔交易、时间和分类/备注，点击可进入交易详情。
- 增加“记账标签分析”入口和卡片，支持收入/支出切换，展示金额、笔数和占比。
- 年度账单入口使用完整宽度的导航行。账单 Sheet 提供年份切换、收入/支出/全部筛选、年度图表和带明确表头的月份数据。
- iPhone 常规 Dynamic Type 使用带明确表头的四列 `Grid`；无障碍字号切换为每月一组带“收入/支出/结余”标签的纵向行。iOS 不设置桌面式最小宽度；macOS 可保留适合窗口的最小尺寸和更高信息密度。

#### Android 同步

- Android 更新为相同的默认月范围、前期汇总、单笔排行和标签分析语义。
- Android 继续使用 Material 3 组件，不复制 iOS 的 Sheet、Liquid Glass 或底部操作样式。

### 4.4 导航、更多页和信息架构

- iOS 主 `TabView` 只保留真正可选择并保持状态的首页、统计、搜索、更多四个目的地。iOS 18+ 使用 `Tab` API，iOS 16–17 保留 availability-gated 的 `tabItem` fallback。
- “记账”改为全局主操作按钮：iOS 26 使用 `tabViewBottomAccessory`；iOS 16–25 在 `TabView` 上使用 `.safeAreaInset(edge: .bottom)` 放置原生 bordered-prominent 按钮和系统 material。按钮打开交易 Sheet，但不伪装成可选择的 Tab。
- 所有程序化 `destination = .transaction` 仍统一走同一新增/编辑入口，保持深层页面发起记账的能力。
- “更多”按数据、账本账户、自动化等分组，使用带模块图标的自适应网格或紧凑导航卡；每项仍是原生 `NavigationLink`，不自绘路由控件。
- 净资产区域复用现有账户余额规则：计入总资产且余额大于零的合计为资产，余额小于零的绝对值合计为负债，净值为全部计入账户余额之和；同时展示三项，不新增账户分类规则。

### 4.5 颜色、无障碍、触控和布局

#### 颜色与状态

- 收入、支出、错误和破坏性操作使用独立语义色，不以主题色替代业务含义。
- 调整收入/支出文本颜色，使普通文字在浅色与深色背景达到至少 4.5:1；图形填充可使用同色的较浅变体，但不可复用为小字号文字。
- 主题选中态前景色按主题背景和当前配色动态选择黑或白，确保所有主题在深浅模式下有足够对比。
- 搜索结果金额始终带 `+`/`−` 和收入/支出可访问文本，不再只靠红绿区分。

#### VoiceOver 与图表

- 应用锁 Toggle 具有明确的“应用锁”标签和当前状态值。
- 详情行将装饰图标隐藏于辅助功能树，并把标题与值组合为一个有意义的元素；按钮和导航行补充适当 hint。
- Swift Charts 不使用 `.accessibilityElement(children: .ignore)` 抹掉系统能力，改用可访问图表描述符。iOS 16 的自定义图形 fallback 同时提供可逐项浏览的数据列表。
- 分类圆环在支持版本使用原生 Chart mark；旧系统保留视觉 fallback，并暴露每个分类的名称、金额和占比。
- 动态菜单和 `ForEach` 使用业务稳定 ID，不使用 `.indices` 或可变 offset。

#### 布局与本地化

- 日历星期标题按 `Calendar.firstWeekday` 旋转，标题顺序与首日空白计算使用同一个 Calendar。
- iOS Sheet 移除 `minWidth: 320...420` 等桌面约束；macOS 尺寸限制放入平台条件分支。
- 首页顶部、统计卡片标题和账单行使用自适应栈、`ViewThatFits` 或合理换行策略，在 320pt 宽度和较大 Dynamic Type 下不裁切关键操作。
- 日期范围保持单行并使用产品指定格式；空间不足时优先缩短格式，不压缩为不可读文本。
- 搜索清除按钮、清除全部、分页/切换图标和所有 Pill 命中区域至少 44pt。

### 4.6 Liquid Glass 与安全性能优化

#### Liquid Glass

- iOS 26 继续使用 `GlassEffectContainer`、`glassEffect` 和原生 toolbar/sheet 结构，所有新 API 都带 `#available` gate。
- `iOSLiquidGlassIconControl` 的视觉图标可以小于 44pt，但外层 frame 和 `contentShape` 固定至少 44pt。
- iOS 16–25 fallback 使用统一 `ButtonStyle` 提供按压透明度或缩放反馈，并尊重 `configuration.isPressed`；不使用不透明自定义背景覆盖系统 bar、sheet 或滚动边缘。
- Pill 的最小高度提升到 44pt，选中与按下状态分别可辨。

#### 性能

- SVG 仍以现有 WebKit 渲染作为无法直接读取资源时的最后 fallback，不引入新依赖。
- 在现有内存缓存和请求合并基础上增加磁盘栅格缓存；缓存键包含资源、颜色、点尺寸和资源版本。磁盘读取不阻塞主线程，WebKit 仍保持有界串行，避免同时创建多个 WebView。
- 对首屏已知图标做小规模预热；预热数量有上限，主题变化只生成当前主题所需变体。
- 把日历摘要字典、分类映射等重复计算移到数据装载或稳定子视图边界；叶子视图尽量接收具体值和操作，而不是为单个字段读取整个 `AppStore`。
- 仅收窄明确热路径，不把 `ObservableObject` 全量迁移或 `AppStore` 拆分作为本次交付条件。
- Widget 读取持久化主题并使用语义色 fallback，不再硬编码与 App 主题冲突的强调色。

## 5. 状态与错误数据流

异步编辑统一遵循以下状态机：

`idle → validating → saving → success` 或 `idle → validating/saving → failure`

- `validating` 和 `saving` 期间保留草稿并阻止重复提交。
- `success` 才允许 dismiss 或进入下一步。
- `failure` 返回编辑态，错误归属到当前表单或字段；全局错误只用于无法归属到单一操作的加载失败。
- 导入行内编辑额外使用 revision，确保完成顺序不影响最终用户输入。
- 日期明细和统计分别维护自己的 loading/error，避免复用全局 `store.loading` 造成错误页面状态。

## 9. 完成标准

本任务只有在以下条件同时满足时才可称为完成：

1. 本文范围内的确定问题均有对应源码修改和测试/检查证据。
