# OmniFlow Progress
## 当前阶段
正在进行完整产品形态的需求澄清。先明确最终想做成的效果、产品边界、数据模型、导入导出规则、分析能力和平台要求，再进入技术选型。
## 已明确需求
### 产品目标
- 开发一个全平台记账 App。
- 最终正式支持 iOS、Android、Web、macOS、Windows。
- 长期目标是替换当前正在使用的「青子记账」App。
- 近期目标需要支持与「青子记账」数据互通，避免迁移期间被单一 App 绑定。
- 当前讨论重点是明确完整产品的最终形态，不以第一版 MVP 限制最终目标。
- 后续可以分阶段实现，但需求澄清阶段需要先把完整效果讨论清楚。
- 最终产品模型支持多账本。
- 第一类核心场景先定义为个人账本。
- 数据模型需要预留家庭共享账本的概念。
### 基础记账能力
- 支持基本记账功能。
- 交易类型只需要支出和收入。
- 支持账户管理。
- 每条收支需要归属到一个账户。
- 账户可用于表示支付宝、微信、建设银行储蓄卡、现金、京东支付等资金来源或去向。
- 具体的分类、预算、多币种等细节尚未澄清。
### 数据分析
- 支持图表展示。
- 支持按周、月、年维度查看统计分析。
- 具体图表类型、指标口径、筛选维度尚未澄清。
### 外部账单导入
计划支持以下来源的账单导入：
- 支付宝
- 微信
- 京东
- 美团
- 银行账单，第一阶段暂时只支持建设银行
导入能力需要支持：
- 解析不同平台的账单文件。
- 明细去重。
- 自动或半自动归类。
- 后续会提供每个支付平台的账单样例文件，格式可能是 CSV 或 JSON 或 XLSX。
### 规则过滤与处理
需要支持用户自定义一系列规则，对导入明细进行处理。
已明确的规则能力：
- 命中关键字后，可以排除该明细。
- 命中关键字后，也可能执行其他操作。
待澄清：
- 规则的匹配字段，例如商户名、商品说明、备注、金额、支付方式、交易对方等。
- 规则动作范围，例如排除、改分类、改账户、加标签、改备注、标记为转账、标记为报销等。
- 因交易类型只保留支出/收入，转账、退款、还款等导入明细需要通过排除、归类或统计口径处理，避免污染真实收支。
- 多条规则同时命中时的优先级。
- 规则是仅用于导入前预处理，还是也能批量作用于已有数据。
### 青子记账互通
需要支持：
- 导入青子记账导出的 JSON 文件。
- 将本项目数据导出为符合青子记账 JSON 格式的数据，方便再导入青子记账。
待澄清：
- 青子记账 JSON 的完整结构和字段含义。
- 本项目内部数据模型与青子记账模型不完全一致时的映射策略。
- 导出时是否要求 100% 可被青子记账无损导入。
## 当前假设
- 第一阶段核心场景是个人账本。
- 最终形态支持多账本，并为家庭共享账本预留模型空间。
- 企业报销、团队财务暂未纳入目标，除非后续明确提出。
- 实现可以分阶段推进，但需求讨论需要覆盖完整产品目标。
- 数据源样例文件会作为解析器设计和测试用例的依据。
- 数据同步方案：本地优先，WebDAV 作为第一优先同步后端。
- 内部同步能力需要抽象为「同步适配器」，未来可扩展 iCloud、坚果云、S3、自建服务等后端。
- 同步到远端的数据默认加密，WebDAV 等远端服务只能看到加密后的数据。
- 本地设备负责解密和使用数据。
这些是假设，不是最终结论，需要继续确认。
## 下一步澄清队列
1. 明确完整产品的使用对象和账本边界：个人记账、家庭共享账本、多人协作、企业/报销场景是否包含。
2. 明确数据存储策略：本地优先、云同步、自建服务、第三方云，还是多端局域网/文件同步。
3. 明确账本核心模型：交易、账户、分类、标签、商户、预算等概念。
4. 明确导入流水线：文件上传、字段映射、预览、规则处理、去重、分类、确认入账。
5. 明确去重策略：平台交易号、时间金额商户组合、手动合并、重复候选提示等。
6. 明确规则系统：匹配条件、动作、优先级、测试预览、作用范围。
7. 明确青子记账互通的兼容等级：临时迁移、长期双写，还是完整双向同步。
8. 明确数据分析口径：收支趋势、分类占比、账户余额、现金流、异常消费、同比环比等。
9. 在需求边界稳定后，再做技术选型。
